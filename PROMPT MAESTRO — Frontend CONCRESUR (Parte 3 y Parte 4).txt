--- Stores ---
comprasStore.js
  state: { categorias: [], proveedores: [],
           compras: [], cuentasPagar: [],
           informeMensual: null }
  actions:
    cargarCategorias()         → GET /compras/categorias
    cargarProveedores()        → GET /compras/proveedores
    crearProveedor(data)       → POST /compras/proveedores
    actualizarProveedor(id, data)
    desactivarProveedor(id)
    cargarCompras(filtros)     → GET /compras/compras
    cargarCompraDetalle(id)    → GET /compras/compras/{id}
    crearCompra(data)          → POST /compras/compras
    registrarAbonoCompra(compra_id, data)
      → POST /compras/compras/{id}/abonos
      → actualizar registro en compras sin refetch total
    cargarCuentasPagar()
    cargarInformeMensual(mes, anio, categoria_id)
      → GET /compras/informe-mensual

  state: { ordenes: [], registros: [],

    cargarOrdenes(filtros)      → GET /produccion/ordenes

ITERACIÓN 4 / 4 — Logística y Financiero
================================================================
MÓDULOS: Kanban Logístico, Cuentas por Cobrar,
         Cuentas por Pagar
ENDPOINTS CONSUMIDOS:
  GET  /logistica/kanban
  GET  /logistica/pedidos/{id}
  PUT  /logistica/pedidos/{id}/estado
  GET  /ventas/cuentas-por-cobrar
  POST /ventas/ventas/{id}/abonos
  GET  /compras/cuentas-por-pagar
  POST /compras/compras/{id}/abonos

--- Nuevas rutas en el router ---
  /logistica/kanban     meta: { permiso: 'logistica.kanban.ver' }
  /financiero/cobrar    meta: { permiso: 'ventas.ventas.ver' }
  /financiero/pagar     meta: { permiso: 'compras.compras.ver' }

--- Stores ---
logisticaStore.js
  state:
    en_espera: []
    en_camino: []
    entregado: []
    pollingInterval: null
    cargando: false
  actions:
    cargarKanban()
      → GET /logistica/kanban
      → poblar las tres listas desde response
      → response esperado: { en_espera:[...], en_camino:[...],
                              entregado:[...] }
    moverEstado(id, nuevo_estado, observaciones)
      Actualización optimista:
        1. Buscar pedido en columna origen
        2. Removerlo de columna origen
        3. Insertarlo en columna destino
        4. Guardar snapshot para revertir
      Luego:
        PUT /logistica/pedidos/{id}/estado
        Si éxito: reemplazar pedido con datos del response
        Si falla: restaurar snapshot + toast.error
      Transiciones válidas:
        'En espera' → 'En camino'
        'En camino' → 'Entregado'
        Cualquier otra → no ejecutar (mostrar toast warning)
    iniciarPolling()
      → setInterval(cargarKanban, 60000)
      → guardar ID en pollingInterval
    detenerPolling()
      → clearInterval(pollingInterval)

financieroStore.js
  state: { cuentasCobrar: [], cuentasPagar: [] }
  actions:
    cargarCuentasCobrar(filtros)
      → GET /ventas/cuentas-por-cobrar
    cargarCuentasPagar(filtros)
      → GET /compras/cuentas-por-pagar
    registrarAbonoCobrar(venta_id, data)
      → POST /ventas/ventas/{venta_id}/abonos
      → actualizar el item en cuentasCobrar con nuevo saldo
      → si saldo_total llega a 0: remover de la lista
    registrarAbonoPagar(compra_id, data)
      → POST /compras/compras/{compra_id}/abonos
      → actualizar el item en cuentasPagar con nuevo saldo
      → si saldo_total llega a 0: remover de la lista

INSTRUCCIONES FINALES PARA EL LLM
================================================================
1. Generar código completo y funcional, sin esqueletos.
2. Usar siempre <script setup> con Composition API en JS plano.
3. Las vistas son delgadas: la lógica va en stores/composables.
4. Llamadas a la API solo en stores o composables,
   nunca directamente en una vista.
5. try/catch en todas las actions de Pinia con toast en el catch.
6. Componentes Base* sin lógica de negocio del dominio.
7. NO avanzar a la siguiente iteración sin cerrar la actual
   al 100% (todos los ítems del checklist marcados).
8. Al finalizar cada iteración entregar:
   - Árbol de archivos creados/modificados
   - Código completo por archivo
   - Rutas del router agregadas en esa iteración
   - Checklist de aceptación marcado
9. Los estados son exactamente los del backend (copiar textual):
   estado_operativo venta:
     'Confirmada' | 'En produccion' | 'Lista' |
     'En camino'  | 'Completada'
   estado_financiero:
     'Pendiente' | 'Pagado'
   estado cotización:
     'Pendiente' | 'Confirmada' | 'Anulada'
   estado orden producción:
     'Pendiente' | 'En proceso' | 'Completada'
   estado logística:
     'En espera' | 'En camino' | 'Entregado'
10. El frontend nunca genera números de documento.
    CMP-XXXX, VTA-XXXX, OP-XXXX los asigna el backend.
11. Las rutas /tablet/produccion y /tablet/jornal son públicas
    (sin requiresAuth) para uso directo desde tablets en planta.
